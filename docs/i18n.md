# Site UI translations (next-intl)

## Locales

- Source files: `messages/en.json`, `messages/es.json`
- Routing: [`i18n/routing.ts`](../i18n/routing.ts) (`localePrefix: "as-needed"` — English URLs have no `/en` prefix; Spanish uses `/es/...`)
- Navigation: import `Link`, `redirect`, `usePathname`, `useRouter` from [`i18n/navigation.ts`](../i18n/navigation.ts) so locale is preserved

## Key naming

- Use **dot namespaces** in the JSON file, e.g. `home.footerAbout`, `about.whatItIsBody`
- Prefer **stable keys** over English copy as the key name
- For **rich text** (links inside a sentence), use `t.rich()` with XML-like tags in the string, e.g. `<marketplace>Marketplace</marketplace>`

## Workflow for translators

1. Developers add or change keys in **`messages/en.json`**
2. Copy new keys to **`messages/es.json`** (or your TMS exports into both files)
3. Run **`npm run check:i18n`** — fails if key sets differ between `en` and `es`
4. Run **`npm run ci`** before merging

Optional: connect **Crowdin / Lokalise / Phrase** to the `messages/` folder for review workflows and machine-translation first drafts.

## Dashboard

The member dashboard and server actions are still **English-first**. Future work can add a `dashboard` namespace and call `getTranslations` / `getLocale()` from server actions where user-facing errors should follow the UI locale.
